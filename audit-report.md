## 二、维度逐项审计

### 1. 产品定位（5.5/10）

**现状**: 定位为"游戏化德州扑克学习应用"，功能覆盖胜率计算、资金管理(BRM)、翻前矩阵、AI 教练四个模块。

**优势**:
- 明确的"学习工具"定位，避开了与专业 Solver 的直接竞争
- 功能组合有差异化（BRM + 翻前矩阵 + AI 教练是独特组合）
- PWA 离线可用，符合学习场景的碎片化需求

**核心问题**:
- 四个模块之间缺乏有机联系，更像是功能堆砌而非产品
- BRM 模块与胜率计算器的数据未打通（需要手动输入 pot/toCall，不能从计算结果自动带入）
- 与竞品 DEEPFOLD（免费版有 40+ 策略分析维度）相比，深度差距巨大
- 目标用户画像模糊：新手嫌功能多，进阶玩家嫌功能浅
- 缺少"练习模式""错题回顾""进度追踪"等学习闭环设计

---

### 2. 算法正确性（6.0/10）

**现状**: 7 选 5 枚举评估牌型，精确计算 + 蒙特卡洛双模式。

**验证通过项**:
- `evaluate5Cards` 的牌型识别逻辑基本正确（高牌→皇家同花顺）
- `evaluate7Cards` 枚举 C(7,5)=21 种组合取最优，逻辑正确
- A-2-3-4-5 小顺（Wheel）特殊处理已包含
- 精确计算 / 蒙特卡洛的阈值切换（200 万组合）合理

**算法缺陷**:
- `calculateEquity` 仅模拟 vs **1 个随机对手**，现实中多为多人底池，此模型严重偏离实战
- `calculateRiskOfRuin` 使用的简化公式 `R = ((1-edge)/(1+edge))^buyIns` 基于独立伯努利试验假设，不适用于扑克（手牌间高度相关、edge 不恒定、有下注决策）
- `getBankrollActionAdvice` 的决策树硬编码阈值（equity < 0.35, < 0.55, < 0.75），未考虑位置、对手形象、牌面结构等关键因素
- `calculateProbabilities` 计算的是"最终牌型概率"而非"胜率"（Equity），命名有误导性
- 蒙特卡洛模拟使用 `Math.random()`，不是密码学安全随机数，对扑克工具可接受但不专业
- `compareHands` 中 kickers 比较时未处理长度不等的情况（虽然当前逻辑下不会触发）

### 3. 性能表现（5.0/10）

**现状**: 纯前端计算，无 Web Worker，主线程阻塞式计算。

**性能瓶颈**:
- **主线程阻塞**: 所有计算（包括蒙特卡洛 20 万次模拟）都在主线程执行，`calculate()` 中的 `setTimeout(..., 100)` 只是视觉欺骗，实际计算仍是同步阻塞
- 在低端手机上，Pre-flop 蒙特卡洛 20 万次 × 21 次 5 牌评估 ≈ 420 万次牌型评估，可能导致数秒卡顿
- `getCombinations` 递归实现，生成 200 万组合数组会直接撑爆内存（虽然 Pre-flop 刚好触发蒙特卡洛规避了，但代码路径存在）
- `evaluate5Cards` 每次调用都对 5 张牌排序（`cards.sort()`），O(n log n) 开销在百万次调用下累积严重
- 没有 Memoization：同一组 7 张牌在不同模拟中重复评估
- BRM 的 Equity 计算（5000 次模拟）与概率计算的 Equity 计算代码重复，没有复用

**内存问题**:
- `calculateExactProbabilities` 中 `getCombinations(deck, remainingToDraw)` 返回完整组合数组，Flop 的 C(47,2)=1,081 个组合无压力，但如果阈值调高会直接 OOM

---

### 4. 架构设计（4.0/10）

**现状**: 零框架、零构建步骤、零依赖的"三无"架构。

**架构问题**:
- **全局命名空间污染严重**: `poker-engine.js` 中所有函数和变量均为全局（`Card`, `SUITS`, `RANKS`, `evaluate5Cards`, `evaluate7Cards`, `calculateEquity`, `PREFLOP_MATRIX`, `AI_COACH_KNOWLEDGE` 等 50+ 个全局符号）
- **无模块化**: 没有 ES Module、没有 IIFE 封装（`i18n.js` 用了 IIFE 但 `poker-engine.js` 没有）、没有命名空间
- **HTML 与 JS 严重耦合**: `index.html` 中内联了 ~500 行 UI 逻辑，与 `poker-engine.js` 没有清晰边界
- **CSS 单文件臃肿**: `style.css` 1412 行，无 BEM/原子化/组件化规范
- **无状态管理**: UI 状态（holeCards, communityCards, cardHistory, selectedSuit）散落在全局变量和 DOM 中
- **无数据层**: 翻前矩阵数据、AI 知识库、翻译文本全部硬编码在 JS 文件中
- **Service Worker 未缓存 i18n.js**: `sw.js` 的 `STATIC_ASSETS` 列表缺少 `/js/i18n.js`，导致离线模式下 i18n 失效

### 5. UI/UX（6.5/10）

**现状**: 绿色赌桌主题，Tesla 极简风格，移动端适配良好。

**设计优势**:
- 主题色系统一（#0d3320 绿底 + #ffd700 金色强调），视觉识别度高
- 响应式布局覆盖 400px 以下到桌面端
- 选牌交互的两步流程（花色→数字）直观
- 概率条动画和阶段指示器有反馈感
- PWA 安装引导按设备类型（iOS/Android/Mac/Windows/Other）区分

**UX 缺陷**:
- **PWA 安装弹窗有 Bug**: `getInstallGuide()` 函数中引用了未定义的变量 `step3`（`index.html:298`），虽然不会抛异常（变量提升为 undefined），但逻辑上永远缺少第 3 步的 iOS/Android 安装指南
- **阶段指示器文本未 i18n**: `updateStageIndicator()` 中硬编码了英文缩写 "Pre-flop"、"Flop" 等，未走翻译系统
- **矩阵展开性能差**: 13×13=169 个 DOM 单元格每次切换位置都全量重建，没有虚拟滚动或复用
- **聊天区没有清空功能**: 消息累积无法清除
- **BRM 输入没有单位提示**: 用户不知道输入的是美元、人民币还是 BB
- **没有历史记录**: 无法回顾之前计算过的手牌
- **没有分享功能**: 无法将一手牌的结果分享给他人
- **暗色/亮色模式切换缺失**: 只有单一主题

### 7. 可扩展性（3.5/10）

**现状**: 单文件、全局变量、硬编码数据。

**扩展障碍**:
- 新增一种语言需要直接修改 `i18n.js` 核心文件（~1000 行），无法插件化加载

- AI 知识库是静态数组，无法热更新或从外部 API 加载
- 没有插件/扩展机制
- 没有 npm 包，无法被其他项目依赖
- 核心算法与 UI 紧耦合，无法单独提取为库使用

### 8. AI/智能模块（4.0/10）

**现状**: 基于规则的 13 条关键词匹配知识库。

**设计判断**:
- 使用规则而非 LLM 是经过调研后的合理选择（通用 LLM 在扑克数学上表现极差）
- 但实现过于简陋

**问题**:
- **知识库仅 13 条**: 覆盖度极低， PokerStrategy 论坛的一个帖子都比这丰富
- **无上下文记忆**: 每次对话独立，无法追问
- **关键词匹配是简单子串查找**: 输入 "fold" 会匹配到 "bankroll" 吗？不会，但 "allin" 和 "all in" 被视为不同词汇
- **建议按钮硬编码中文**: `sendSuggestion('资金只有20个买入，我该用什么策略？')` 在英文模式下仍以中文发送
- **没有模糊匹配/同义词处理**: "下注"和"投注"被视为不同词
- **没有对话状态机**: 无法引导用户完成一个多轮策略分析
- **AI 教练名称/人格缺失**: 只是一个回复机器人，没有角色设定

### 12. 国际化（7.0/10）

**现状**: 中英双语，基于 `data-i18n` 属性的 DOM 翻译系统。

**优势**:
- `i18n.js` 采用 IIFE 封装，避免全局污染
- 支持插值参数（`{0}` 替换）
- `data-i18n`/`data-i18n-placeholder`/`data-i18n-title` 属性覆盖全面
- 语言切换持久化到 localStorage
- 触发 `languageChanged` 自定义事件，支持动态内容重渲染

**问题**:
- **PWA manifest.json 硬编码 `lang: "zh-CN"`**，无法随语言切换
- **HTML lang 属性切换不彻底**: `<html lang="zh-CN">` 初始写死，切英文后变为 `lang="en"`，但某些屏幕阅读器可能不识别
- **翻译键命名不一致**: 有的是 camelCase（`brm.brm.safe`），有的是点分隔层级（`handNames.0`），有的是扁平（`app.title`）
- **缺少 RTL 支持**: 阿拉伯语/希伯来语等 RTL 语言无法正确渲染
- **数字/日期/货币未本地化**: 百分比、千分位分隔符等在不同语言下的显示格式未处理
- **i18n.js 文件过大**: 中英各 ~500 行翻译放在一个文件中，新增语言会导致文件膨胀

### 15. 代码质量（4.5/10）

**代码审查发现的具体问题**:

**A. index.html**
1. `getInstallGuide()` 引用未定义变量 `step3`（line ~298）
2. `originalTexts` 对象在 `updateStageIndicator()` 中硬编码英文，未走 i18n
3. `renderRankPicker()` 中 `RANKS_ORDER` 硬编码为 `['A','K',...'2']`，与 `poker-engine.js` 中的 `RANKS` 不一致（后者是 `['2','3',...'A']`）
4. `escapeHtml()` 每次调用都创建新的 `<div>`，应复用或改用 replace
5. 全局变量 `holeCards`, `communityCards`, `selectedSuit`, `cardHistory` 等无命名空间保护
6. `updateRankButtons()` 中内层循环 `for (const s of SUITS)` 每次点击都遍历 4 个花色，效率低
7. `calculateBRM()` 中 `alert()` 使用原生弹窗，体验差

**B. js/poker-engine.js**
8. 全局命名空间污染：50+ 个全局符号
9. `HAND_NAMES` 使用 Proxy，在 `calculateProbabilities` 中通过 `HAND_NAMES[rank]` 访问，但 `rank` 是数字，`Proxy.get` 的 `prop` 是字符串，依赖隐式类型转换
10. `getBankrollActionAdvice` 函数过长（~200 行），单一职责原则严重违反
11. `PREFLOP_MATRIX` 数据缺失大量手牌键：虽然默认值 'F' 兜底，但数据完整性无法验证
12. `ACTION_MEANINGS` Proxy 的 fallback 路径和 `t()` 路径返回的对象结构不一致（fallback 有 `text/color/bg/desc`，`t()` 路径只有 `text`）
13. `calculateEquity` 的 5000 次模拟是硬编码，无参数化
14. `calculateMonteCarloProbabilities` 中 Fisher-Yates 洗牌每次迭代都执行，应使用更高效的采样方法
15. `evaluate5Cards` 对输入数组原地排序（`cards.sort()`），产生副作用——虽然调用方传递的是新数组，但这是一个隐患
16. 缺少输入校验：`evaluate7Cards` 不检查输入是否为 7 张牌
17. `compareHands` 的 kickers 比较在长度不等时行为未定义

**C. js/i18n.js**
18. 翻译键散落在两个巨大对象中，没有按模块/领域分组文件
19. 没有键缺失检测机制：如果某个键在一种语言中缺失，静默回退到另一种语言
20. `applyTranslations()` 每次遍历整个 DOM，语言切换时有性能开销

**D. css/style.css**
21. 没有 CSS 变量覆盖策略，主题切换只能靠替换整个文件
22. `!important` 使用过多（在 `prefers-reduced-motion` 中），破坏层叠
23. 缺少 CSS 架构规范（如 BEM），类名有时语义化有时不
24. 媒体查询重复：`(max-width: 600px)` 出现多次，应合并

**E. sw.js**
25. Cache-First 策略但缺少缓存更新机制：用户可能永远看不到新版本
26. 缺少 `i18n.js` 缓存，离线模式翻译失效
27. `manifest.json` 未被缓存路径正确匹配（请求路径可能是 `./manifest.json` 但缓存键是 `/manifest.json`）

**F. test/test-engine.cjs**
28. 测试用例不覆盖所有牌型踢脚比较
29. 缺少对 `calculateProbabilities` 返回结果结构的断言
30. 使用 `JSON.stringify` 比较对象，对浮点数不友好
